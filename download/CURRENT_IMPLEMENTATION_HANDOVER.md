# 1. Executive Summary

The **Ibn Hayan Healthcare Operating System** (`ibn-hayan-healthcare-os`) is a pnpm-managed TypeScript monorepo that implements the foundational spine of a multi-tenant, bilingual (Arabic-first) healthcare platform. The repository is genuinely runnable in development: a pnpm workspace, a Next.js 16 web client, a NestJS 11 API, three shared packages (`contracts`, `domain`, `configuration`, `observability`, `testing`), a PostgreSQL 17 + Prisma 7 persistence layer, and three reviewed SQL migrations are all in place and committed.

**What is genuinely implemented today (verified in code):**

1. **Health endpoint** — `GET /api/v1/health` returns a constant `{status, service, version}` JSON body that is validated against a shared Zod contract. OpenAPI documentation is served at `/api/docs` and `/api/docs-json` outside production.
2. **Tenancy foundation** — `Tenant`, `Organisation`, `Facility` Prisma models with composite foreign keys (added through reviewed raw SQL because Prisma 7 cannot express them) and `ON DELETE RESTRICT` on every relationship.
3. **Identity & session foundation** — `User`, `LocalCredential`, `TenantMembership`, `AuthSession` models. Argon2id password hashing via the `argon2` package. SHA-256 hashed opaque session tokens. HttpOnly + SameSite=Lax cookies (Secure in production).
4. **Authentication API** — `POST /api/v1/auth/login`, `GET /api/v1/auth/session`, `GET /api/v1/auth/csrf`, `POST /api/v1/auth/logout`. Login is throttled (10 attempts / 60 s per IP via `@nestjs/throttler`). Login failures return the same generic 401 for unknown email, wrong password, disabled user, and no active membership.
5. **Session tenant-context API** — `GET /api/v1/context`, `PUT /api/v1/context/tenant`, `DELETE /api/v1/context/tenant`. Selection is by `TenantMembership` ID, never by raw Tenant ID. A composite database foreign key structurally rejects cross-user context selection.
6. **Premium landing + login surface** — `apps/web/src/app/page.tsx` renders a bilingual Arabic-first marketing landing page with an integrated login card. The page is the Batch 6 redesign committed in `de2d7ed`.
7. **Authenticated dashboard shell** — `/dashboard` loads the session, loads the user's available tenant memberships, allows selection/clearing of the active workspace, displays the user's account card, and supports logout.
8. **Shared contracts** — `@ibn-hayan/contracts` (Zod schemas for `HealthResponse`, `LoginRequest`, `SessionResponse`, `CsrfResponse`, `LogoutResponse`, `ContextResponse`, `SelectTenantContextRequest`, `ClearTenantContextResponse`, `AuthErrorResponse`) is consumed by both the API and the web client.
9. **Domain layer** — `@ibn-hayan/domain` declares pure-TypeScript bounded-context types and repository ports for `tenancy` and `identity`. The API's persistence layer implements those ports via Prisma-backed adapters.
10. **Test suite** — Vitest with separate configs for unit, integration, database, auth, and context tests. Database tests boot a disposable PostgreSQL 17 cluster and exercise real migrations.

## 7.5 Authentication-related screens

The login screen is `apps/web/src/components/marketing/login-panel.tsx` (rendered inside `Hero` on both `/` and `/login`). There is **no separate password-reset screen, no MFA screen, no signup screen, no email-verification screen** — login is the only auth-related surface.

The dashboard page (`/dashboard`) performs the authenticated-side auth flow:
1. On mount, calls `getSession()`. On 401, redirects to `/login`.
2. On success, calls `getContext()` to load the available workspaces and the active workspace.
3. Before any state-changing call (logout, select-context, clear-context), calls `getCsrfToken()` to obtain a fresh CSRF token.
4. Logout: `getCsrfToken()` → `logout(token)` → redirect to `/login`.
5. Select workspace: `getCsrfToken()` → `selectTenantContext(membershipId, token)` → update local state.
6. Clear workspace: `getCsrfToken()` → `clearTenantContext(token)` → re-fetch `getContext()`.

## 7.8 Tenant selection

Tenant selection lives entirely in `DashboardPage`'s `handleSelectContext` handler:
1. `getCsrfToken()` — fetches a fresh CSRF token.
2. `selectTenantContext(selectedMembershipId, token)` — `PUT /api/v1/context/tenant` with the membership ID in the body and the CSRF token in the `X-CSRF-Token` header.
3. On success, updates local `context` state and `selectedMembershipId`.
4. On failure, displays a generic error message.

The user selects by **membership ID**, never by Tenant ID. The list of available memberships is rendered as a radio group; only memberships whose status is `active` and whose Tenant is also `active` are shown.

## 7.9 Localization & Arabic/RTL support

- The `<html>` element defaults to `lang="ar" dir="rtl"`.
- The `LanguageProvider` updates `document.documentElement.lang` and `document.documentElement.dir` when the user toggles.
- Every page-level component consumes `useLanguage()` to get `{lang, dir}`.
- Every text-bearing component looks up copy from a hardcoded bilingual constant (`LANDING_COPY` for landing, `DASHBOARD_COPY` for dashboard).

This is sufficient for the current two-language, two-page surface; it will need to be revisited if the product grows to many pages or many languages.

## 7.10 API client implementation

The API client lives in `apps/web/src/lib/api/`:

- `api-url.ts` — `getApiBaseUrl()` reads `process.env.NEXT_PUBLIC_API_BASE_URL` with a development fallback of `http://localhost:3001/api/v1`. The URL is normalised (trailing slashes removed). `joinUrl(base, path)` joins a base URL with a path.
- `api-error.ts` — defines `ApiError` and four error categories: `NETWORK_ERROR`, `HTTP_ERROR`, `INVALID_JSON`, `CONTRACT_INVALID`. Every failure mode of every API client is classified into one of these categories. Raw error details (network messages, response bodies, stack traces) are **never** exposed to the UI.
- `health.client.ts` — `fetchHealth()` → `GET /api/v1/health`. Parses the response through `HealthResponseSchema`.
- `auth/auth.client.ts` — `login()`, `getSession()`, `getCsrfToken()`, `logout()`. All use `credentials: 'include'`. Every successful response is parsed through the corresponding shared Zod schema. Every failure returns `{ok: false, error: ApiError}` — the client **never throws**.
- `context/context.client.ts` — `getContext()`, `selectTenantContext()`, `clearTenantContext()`. Same shape as the auth client.

The clients use the platform `fetch` API — no Axios, no ky, no got. The CSRF token is held in component memory only; it is **never** persisted to `localStorage`, `sessionStorage`, `IndexedDB`, or a readable cookie.

## 7.11 Session handling

Sessions are entirely server-side. The browser holds only the opaque session token in an HttpOnly cookie (`ibn_hayan_session`). The web client never reads the cookie value (it cannot — the cookie is HttpOnly). It relies on the cookie being sent automatically with every `credentials: 'include'` request.

The web client's view of the session is the `SessionResponse` JSON body returned by `/auth/session`:
```ts
{
  user: { id, email, displayName, status },
  memberships: Array<{ id, tenantId, tenantSlug, tenantDisplayName, status }>,
  activeTenantContext: { membershipId, tenantId, tenantSlug, tenantDisplayName } | null,
  expiresAt: string  // ISO 8601
}
```

The client does NOT track session expiry client-side; it relies on the next `getSession()` call returning 401 to discover expiry. There is no proactive refresh.

## 7.12 Error handling

Three layers:

1. **Network layer** (`api-error.ts`): Every `fetch` exception is wrapped in `networkError(error)`. Every non-2xx HTTP response is wrapped in `httpError(statusCode)`. Every JSON parse failure is wrapped in `invalidJsonError(error)`. Every Zod validation failure is wrapped in `contractInvalidError(error)`. None of these expose raw error details to the UI.
2. **Component layer** (`LoginPanel`, `DashboardPage`): Each handler inspects `result.ok` and `result.error.category` (or `result.error.statusCode`). The user-facing message is a generic bilingual string. Raw error messages are never displayed.
3. **Page layer**: `/dashboard` redirects to `/login` when `getSession()` returns 401. No error page is shown for 401; the user simply returns to the landing page.

There is no `error.tsx`, no `global-error.tsx`, no Next.js error boundary. The `LoginPanel` and `DashboardPage` use inline `StatusMessage` components to display errors.

## 8.6 Guards, middleware, interceptors, filters

## 8.11 CSRF protection

Two layers:
1. **Origin verification** (primary): Every state-changing auth/context endpoint (`POST /auth/login`, `POST /auth/logout`, `PUT /context/tenant`, `DELETE /context/tenant`) verifies the `Origin` header against `WEB_ORIGIN`. Missing or disallowed Origin returns a generic 403.
2. **CSRF token** (secondary): The web client calls `GET /auth/csrf` to obtain a raw token. The token is held in component memory only. The client sends the token in the `X-CSRF-Token` header on logout and on context mutations. The server hashes the submitted token and compares the hash against the stored hash using `timingSafeEqual`.

The CSRF token is **session-bound**: a token issued for one session cannot be used with another session's request. The hash is stored in an in-memory `Map<SessionId, string>` in `CsrfService`.

# 9. Authentication and Session Flow

This section traces the authentication flow end-to-end. Every file referenced below was read in full during this audit.

## 9.1 Login request

The browser POSTs to `https://<api-origin>/api/v1/auth/login` with:
- Headers: `Content-Type: application/json`, `Accept: application/json`, `Origin: https://<web-origin>`.
- Body: `{ "email": "...", "password": "..." }`.
- Credentials: `include` (so the response's `Set-Cookie` is stored).

The web client's `login()` function (`apps/web/src/lib/api/auth/auth.client.ts`) validates the input through `LoginRequestSchema.safeParse(input)` before sending. If validation fails, the client returns `{ok: false, error: contractInvalidError(...)}` without making a network request.

## 9.2 Credential verification

`AuthController.login` (`apps/api/src/modules/auth/auth.controller.ts`):
1. Parses the body through `LoginRequestSchema.safeParse(body)`. On failure, throws `invalidCredentials()` — a generic 401 that does **not** reveal whether the email exists.
2. Reads `Origin` header and `WEB_ORIGIN` config. Calls `AuthService.login({email, password, origin, webOrigin})`.

`AuthService.login` (`apps/api/src/modules/auth/auth.service.ts`):
3. **Origin check first** — `isOriginAllowed(origin, webOrigin)`. Throws `originDisallowed()` (403) if Origin is missing or not in the allowed list. This check runs **before any credential lookup** so that the response does not reveal whether an account exists.
4. Normalises the email (`trim().toLowerCase()`).
5. Looks up the user by `normalisedEmail` via `UserRepository.findByNormalisedEmail()`. If no user, throws `invalidCredentials()`.
6. If `user.status === 'disabled'`, throws `invalidCredentials()`.
7. Calls `PasswordService.verifyForUser(user.id, password)`. If false, throws `invalidCredentials()`.
8. Loads the user's memberships via `TenantMembershipRepository.listForUser(user.id)`. Filters to `status === 'active'`. If the filtered list is empty, throws `invalidCredentials()`.

All four failure cases (unknown email, wrong password, disabled user, no active membership) throw the **same exception**. The caller cannot distinguish them.

## 9.3 Password hashing and comparison

`PasswordService` (`apps/api/src/modules/auth/password.service.ts`):
- Hashing: `argon2.hash(password, { type: argon2.argon2id, memoryCost, timeCost, parallelism })` — returns a PHC-string-formatted hash (e.g. `$argon2id$v=19$m=65536,t=3,p=4$<salt>$<hash>`).
- Verification: `argon2.verify(hash, password)` — uses argon2's internal constant-time comparison. If the hash is malformed or argon2 throws, the method returns `false` (rather than throwing) so the caller can treat it identically to a wrong password.
- Parameters: read from `process.env.ARGON2_MEMORY_COST` (default 65536 KiB), `ARGON2_TIME_COST` (default 3), `ARGON2_PARALLELISM` (default 4). The `readPositiveIntFromEnv` helper falls back to the default if the env var is missing or non-numeric.

The hash is **never** exposed through the domain `User` type or through any API response. It lives only on the `local_credentials` table, accessed only by `LocalCredentialService` (an infrastructure-only service).

## 9.4 Session creation

Back in `AuthService.login`:
9. `now = new Date()`. `expiresAt = new Date(now.getTime() + SESSION_ABSOLUTE_TTL_MS)` (12 hours).
10. `const { raw, hash } = generateSessionTokenAndHash(this.sessionTokens)` — generates a 32-byte random token (base64url, 43 chars) and its SHA-256 hash (64 hex chars).
11. `await this.sessions.create({ userId, tokenHash: hash, expiresAt, lastSeenAt: now })` — persists the session row. The raw token is **never** persisted.

## 9.5 Cookie creation

Back in `AuthController.login`:
12. `const isProduction = process.env['NODE_ENV'] === 'production'`.
13. `const maxAge = result.expiresAt.getTime() - Date.now()` (ms until expiry).
14. `res.cookie('ibn_hayan_session', result.rawToken, buildSessionCookieOptions(isProduction, maxAge))` — sets the HttpOnly cookie.

`buildSessionCookieOptions` (`apps/api/src/modules/auth/auth.cookies.ts`):
```ts
{
  httpOnly: true,
  secure: isProduction,  // false in dev, true in prod
  sameSite: 'lax',
  path: '/',
  maxAge: maxAgeMs,
}
```

No `domain` is set. The cookie is bound to the exact origin that set it.

## 9.6 CSRF handling

CSRF is handled by `CsrfService` (`apps/api/src/modules/auth/csrf.service.ts`):
- **Token issuance**: `GET /api/v1/auth/csrf` (authenticated). `CsrfService.generate(sessionId)`:
  1. `randomBytes(32)` → base64url (43 chars).
  2. `hash = SHA-256(raw)` (64 hex chars).
  3. `this.tokens.set(sessionId, hash)` — stores the hash in an in-memory `Map<SessionId, string>`. If a token already exists for the session, it is **replaced** and the previous token is invalidated immediately.
  4. Returns the raw token in the JSON body: `{ token: "..." }`.
- **Token verification**: `CsrfService.verify(sessionId, submittedToken)`:
  1. `storedHash = this.tokens.get(sessionId)`. If undefined, returns false.
  2. `submittedHash = SHA-256(submittedToken)`.
  3. `timingSafeEqual(Buffer.from(submittedHash, 'hex'), Buffer.from(storedHash, 'hex'))` — constant-time comparison.
- **Token invalidation**: `CsrfService.invalidate(sessionId)` — `this.tokens.delete(sessionId)`. Called at logout.

The web client holds the raw CSRF token in component memory only (never in `localStorage`, `sessionStorage`, `IndexedDB`, or a readable cookie). It sends the token in the `X-CSRF-Token` header on logout and on context mutations.

## 9.7 User retrieval

`AuthService.getSessionFromCookie(cookieValue)`:
1. If the cookie is missing or empty, returns null.
2. `tokenHash = SHA-256(cookieValue)`.
3. `session = await SessionRepository.findActiveByTokenHash(tokenHash, now)` — looks up the session row by `token_hash` (unique index). Returns null if the row does not exist, is revoked (`revokedAt !== null`), or is expired (`expiresAt <= now`).
4. `user = await UserRepository.findById(session.userId)`. Returns null if the user does not exist or is disabled.
5. Loads the user's memberships. Filters to active. If empty, returns null.
6. Decides whether to touch, rotate, or do nothing (see 9.9).
7. Returns `{ session, user, memberships, expiresAt, rotatedRawToken }`.

## 9.8 Tenant selection

The user selects a Tenant by calling `PUT /api/v1/context/tenant` with `{ membershipId: "..." }` in the body. `SessionContextService.selectContext(cookieValue, membershipId)`:
1. Authenticates the session via `AuthService.getSessionFromCookie(cookieValue)`. Returns null if invalid.
2. Looks up the membership by ID via `TenantMembershipRepository.findById(membershipId)`.
3. If the membership is null, **OR** belongs to a different user, **OR** has status !== 'active', throws `contextSelectionForbidden()` — a generic 403 that does not reveal which condition failed.
4. Looks up the Tenant via `TenantRepository.findById(membership.tenantId)`. If null or status !== 'active', throws `contextSelectionForbidden()`.
5. `await SessionRepository.setActiveTenantMembership(session.id, membershipId, now)` — persists the active membership on the session row.
6. Returns the full `ContextResponse` (options + active).

The database composite foreign key `auth_sessions(active_tenant_membership_id, user_id) → tenant_memberships(id, user_id)` (added by the reviewed raw SQL supplement in `20260718220000_session_tenant_context`) is the **structural backstop** — even if the application layer had a bug, the database would reject a cross-user context selection.

## 9.9 Session refresh or expiration

`AuthService.getSessionFromCookie` performs two maintenance operations:
- **Touch**: If `now - session.lastSeenAt >= SESSION_IDLE_TOUCH_INTERVAL_MS` (5 min), updates `lastSeenAt = now`. No token replacement.
- **Rotate**: If `now - session.rotatedAt >= SESSION_ROTATION_INTERVAL_MS` (30 min) (or `rotatedAt` is null and `now - createdAt >= 30 min`), generates a new token, replaces the stored hash via `SessionRepository.rotateToken(session.id, newHash, now)`, and returns the new raw token to the caller. The controller sets a new cookie with the new token. The previous token is **immediately invalidated** (the hash is replaced in the row).

The session's `expiresAt` is **never extended**. The 12-hour absolute TTL is fixed at creation. The session expires when `now >= expiresAt`; the next `getSession()` call returns null and the web client redirects to `/login`.

There is no refresh-token mechanism. There is no sliding-window expiration.

## 9.10 Logout

`AuthController.logout`:
1. Reads `Origin`, `X-CSRF-Token` header, and `ibn_hayan_session` cookie.
2. Calls `AuthService.logout({ cookieValue, origin, csrfToken, webOrigin })`:
   a. Verifies Origin. Throws `originDisallowed()` (403) on mismatch.
   b. Validates the cookie. If empty, throws `csrfInvalid()` (403).
   c. Looks up the session by token hash. If not found, throws `csrfInvalid()` (403) — this is intentional: logout without a session still requires a valid CSRF token to prevent logout CSRF.
   d. Verifies the CSRF token via `CsrfService.verify(session.id, csrfToken)`. Throws `csrfInvalid()` on mismatch.
   e. `await SessionRepository.revoke(session.id, now)` — sets `revokedAt = now`.
   f. `CsrfService.invalidate(session.id)` — removes the CSRF hash from the in-memory map.
3. `res.clearCookie('ibn_hayan_session', buildSessionCookieClearOptions(isProduction))` — clears the cookie (same options as set, except `maxAge: 0`).
4. Returns `{ ok: true }`.

## 9.14 Brute-force / rate-limit protections

- `@nestjs/throttler` is installed. `ThrottlerGuard` is registered as `APP_GUARD` in `AuthModule`.
- `ThrottlerModule.forRoot([{ name: 'default', ttl: 60_000, limit: 1_000 }])` — permissive default (1000 req/60s) for all routes.
- `@Throttle({ default: { limit: 10, ttl: 60_000 } })` on `AuthController.login` — 10 login attempts per 60s per IP.
- `@SkipThrottle()` on `HealthController` — health is exempt from throttling (so a flood of login attempts cannot make the liveness probe return 429).
- The throttle is **per-IP**, not per-email. A distributed attack from many IPs could still attempt many passwords per email. The auth service comment explicitly notes: "Per-email throttling is a future batch concern (it requires persisting attempt counts and would itself need to avoid leaking account existence)."

There is **no account lockout** after N failures, **no progressive backoff**, **no CAPTCHA**, **no MFA**, **no password-reset flow**, **no email-verification flow**.

## 9.15 Authorization guards

There are **no NestJS guards** for auth. Authentication is performed manually inside each controller method via `AuthService.getSessionFromCookie()` + a null-check that throws `sessionRequired()`. This is a deliberate choice documented in the auth module comments: there is no `@UseGuards(AuthGuard)` pattern, no `@CurrentUser()` decorator, no request-scoped user object.

## 9.16 Unresolved security risks

| Risk | Severity | Evidence |
|---|---|---|
| CSRF hash is in-process only | Medium | `CsrfService.tokens` is a `Map<SessionId, string>`. A multi-process deployment (e.g. PM2 cluster, Kubernetes replicas) requires sticky-session routing or the CSRF verification will fail when a request hits a different process. The code comments acknowledge this and defer the fix to a future batch (database persistence or HMAC-derived token). |
| Throttle is per-IP, not per-email | Medium | Distributed brute-force attacks are not mitigated. |
| No RLS policies | Medium | The schema comments explicitly say: "No row-level security policies exist yet. | No audit trail | High | No audit logging exists. Failed logins, successful logins, logouts, context selections, and context clears are **not** recorded anywhere. The `AUDIT.md` documentation describes a comprehensive audit primitive, but none of it is implemented. |
| No account lockout | Low | The 10/60s throttle is the only brute-force defence. |
| No MFA | Low (intentional) | Out of scope for the current batch. |

# 10. Multi-Tenancy Implementation

## 10.1 What multi-tenancy currently means in the code

Multi-tenancy in the current implementation means: **a User can be a member of multiple Tenants, and a session can have at most one active Tenant context selected from the user's memberships.** That is the entire surface. There are no tenant-scoped business objects (patients, appointments, etc.), no tenant-scoped API endpoints beyond the context-selection endpoints, and no tenant-scoped authorization checks beyond the membership-ownership check.

## 10.2 Tenant data model

The schema (`apps/api/prisma/schema.prisma`) defines a three-level tenancy hierarchy:

```
Tenant  (top-level isolation boundary)
  └── Organisation  (second level)
        └── Facility  (third level)
```

- `Tenant` — fields: `id` (UUID), `slug` (unique, VARCHAR(80)), `displayName` (VARCHAR(200)), `status` (`active` | `suspended`), `createdAt`, `updatedAt`.
- `Organisation` — fields: `id`, `tenantId`, `code` (VARCHAR(50)), `displayName`, `status` (`active` | `inactive`), `createdAt`, `updatedAt`. Unique constraint on `(tenantId, code)` — the same code can be reused across tenants. Composite unique on `(tenantId, id)` to support the composite FK from `facilities`.
- `Facility` — fields: `id`, `tenantId`, `organisationId`, `code`, `displayName`, `status` (`active` | `inactive`), `createdAt`, `updatedAt`. Unique constraint on `(tenantId, organisationId, code)`. `tenantId` is persisted on the facility row so that tenant-scoped queries do not need to join through `organisations`.

All three use `ON DELETE RESTRICT` on every foreign key. Deleting a Tenant with active Organisations is rejected; deleting an Organisation with active Facilities is rejected.

## 10.3 Organisation and Facility relationship

A Facility belongs to exactly one Organisation, which in turn belongs to exactly one Tenant. The `tenantId` column on `facilities` is denormalised for query efficiency — it must match the Organisation's `tenantId`. This invariant is enforced by a **composite foreign key**:

```sql
ALTER TABLE "facilities"
  ADD CONSTRAINT "facilities_tenant_id_organisation_id_fkey"
  FOREIGN KEY ("tenant_id", "organisation_id")
  REFERENCES "organisations" ("tenant_id", "id")
  ON DELETE RESTRICT
  ON UPDATE RESTRICT;
```

This composite FK is added by the reviewed raw SQL supplement in `20260718170628_tenancy_foundation/migration.sql` because Prisma 7 cannot express composite foreign keys in its schema language. The single-column FK `facilities.organisation_id → organisations.id` is also declared (via Prisma) for query-builder convenience, with `onDelete: Restrict`.

## 10.4 Membership model

`TenantMembership` is the link between a `User` and a `Tenant`. A User may hold at most one membership per Tenant (enforced by the unique constraint on `(tenantId, userId)`). A User's set of memberships determines the Tenants they may operate within.

Fields: `id`, `tenantId`, `userId`, `status` (`active` | `suspended`), `createdAt`, `updatedAt`.

**No role or permission columns exist on `TenantMembership`.** The schema comments explicitly say: "The role/permission catalogue is deferred to a subsequent batch."

A composite unique constraint on `(id, userId)` exists specifically to support the composite foreign key from `auth_sessions`:

```sql
CREATE UNIQUE INDEX "tenant_memberships_id_user_id_key"
  ON "tenant_memberships"("id", "user_id");
```

## 10.5 How the current tenant is selected

The user selects a Tenant by calling `PUT /api/v1/context/tenant` with a `membershipId` in the body. The selection is persisted as `auth_sessions.active_tenant_membership_id` (a nullable UUID column added by the `20260718220000_session_tenant_context` migration).

Selection is **by membership ID, never by raw Tenant ID**. The server verifies that the membership:
1. Exists.
2. Belongs to the authenticated user (`membership.userId === session.userId`).
3. Has status `active`.
4. Belongs to a Tenant whose status is `active`.

If any check fails, the server throws `contextSelectionForbidden()` — a generic 403 that does **not** reveal which condition failed (so the caller cannot probe which membership IDs exist for other users).

The active context can be cleared by `DELETE /api/v1/context/tenant`, which sets `active_tenant_membership_id = null`.

## 10.6 How tenant context reaches backend requests

Today, the only consumer of the active Tenant context is the `GET /api/v1/context` and `PUT /api/v1/context/tenant` and `DELETE /api/v1/context/tenant` endpoints themselves. There are **no business endpoints** (no `/patients`, no `/appointments`, etc.) that consume the active context.

When business endpoints are added in future batches, they will need to:
1. Authenticate the session via `AuthService.getSessionFromCookie()`.
2. Read `session.activeTenantMembershipId`.
3. If null, return a 403 or 412 ("select a workspace first").
4. If non-null, scope every database query by the selected Tenant.

The infrastructure for this is in place: the `auth_sessions` row carries `active_tenant_membership_id`, and the composite FK guarantees it belongs to the session's user.

## 10.7 Tenant isolation enforcement

Today, tenant isolation is enforced at **two layers**:

1. **Application layer**: `SessionContextService.selectContext()` explicitly checks `membership.userId === authResult.user.id` before setting the active context. A cross-user selection throws `contextSelectionForbidden()`.
2. **Database layer**: The composite foreign key `auth_sessions(active_tenant_membership_id, user_id) → tenant_memberships(id, user_id)` (added by the reviewed raw SQL supplement in `20260718220000_session_tenant_context`) **structurally rejects** a cross-user context selection at the database level. Even if the application layer had a bug, the database would reject the insert.

## 10.8 Database query scoping

There is **no tenant-scoping middleware** and **no tenant-scoping Prisma extension**. Every query in the codebase is scoped explicitly by the application layer:
- `TenantMembershipRepository.listForUser(userId)` — returns only the user's memberships.
- `SessionRepository.findActiveByTokenHash(tokenHash, now)` — looks up by token hash, returns the session row (which carries `userId` and `activeTenantMembershipId`).
- `TenantRepository.findById(tenantId)` — looks up a single tenant by ID. There is no `listAll()` method.

When business endpoints are added, each one will need to scope its queries by `session.activeTenantMembershipId` → `membership.tenantId`. There is no shared helper for this today.

## 10.9 Cross-tenant protections

| Protection | Layer | Status |
|---|---|---|
| Session cannot reference another user's membership | Database (composite FK) | **Implemented** |
| Context selection requires active membership owned by the session user | Application | **Implemented** |
| Suspended membership cannot be selected as active context | Application | **Implemented** |
| Membership under a suspended Tenant cannot be selected | Application | **Implemented** |
| Tenant deletion is restricted while Organisations exist | Database | **Implemented** |
| Organisation deletion is restricted while Facilities exist | Database | **Implemented** |
| Tenant deletion is restricted while memberships exist | Database | **Implemented** |
| Cross-tenant Facility-Organisation attachment is rejected | Database (composite FK) | **Implemented** |
| Row-Level Security policies | Database | **NOT implemented** (deferred per schema comments) |
| Database-role separation per tenant | Database | **NOT implemented** |
| Tenant-scoped query middleware | Application | **NOT implemented** (no business endpoints exist yet) |
| Tenant-scoped audit trail | Application + Database | **NOT implemented** (no audit at all) |

## 10.10 Administrator behaviour

There is **no administrator role** and **no super-admin console**. The `Tenant` model has no `owner` field. The `User` model has no `isSuperAdmin` flag. Every user is a normal user with zero or more memberships.

A user with multiple memberships can switch between tenants by calling `PUT /api/v1/context/tenant` with a different `membershipId`. There is no "all tenants" view.

## 10.11 Missing isolation controls

- **RLS policies**: The schema explicitly defers RLS to a future batch. Today, a SQL query run directly against the database (e.g. by a DBA or a backup restore) can read any tenant's data. The application layer is the only enforcement.
- **Per-tenant database roles**: Not implemented. The API uses a single `DATABASE_URL` with a single PostgreSQL role.
- **Tenant-scoped query helper**: No shared utility exists. Future business endpoints will need to repeat the "resolve tenant from session → scope query" pattern manually.
- **Audit trail**: No audit logging exists. Cross-tenant access attempts are not recorded.

## 10.12 Differences between architecture documentation and actual implementation

| Documented (architecture docs) | Implemented (code) | Gap |
|---|---|---|
| 18 bounded contexts (BC1–BC18) including tenancy, identity, audit, configuration, feature-flags, patients, appointments, billing, etc. | 2 bounded contexts (`tenancy`, `identity`) in `packages/domain/src/`. | 16 bounded contexts are documented but not implemented. |
| Multi-tenant with RLS policies and database-role separation | Multi-tenant with application-layer checks + composite FKs. No RLS, no role separation. | RLS and role separation are explicitly deferred. |
| Comprehensive audit primitive (every consequential action audited) | No audit at all. | Major gap. |
| Role/permission catalogue on `TenantMembership` | `TenantMembership` has only `id, tenantId, userId, status, createdAt, updatedAt`. | The role/permission catalogue is deferred. |
| Feature-flag service (BC18) | No feature-flag service. `packages/configuration` is empty. | Major gap. |
| Configuration service (BC19) | No configuration service. | Major gap. |
| Workflow engine | No workflow engine. | Major gap. |
| Offline-first synchronisation | No offline-first. The web client is a thin client that requires the API to be reachable. | Major gap. The web client has no service worker, no IndexedDB, no sync queue. |

## 11.10 Cascade behaviour

**No cascading deletes anywhere.** Every foreign key uses `ON DELETE RESTRICT`. The schema comments explicitly state: "No cascading deletes. All foreign keys use ON DELETE RESTRICT." ## 11.15 Naming conventions

- **Prisma model names**: PascalCase (e.g. `TenantMembership`, `AuthSession`).
- **Database table names**: snake_case plural (e.g. `tenant_memberships`, `auth_sessions`), via `@@map`.
- **Database column names**: snake_case (e.g. `tenant_id`, `created_at`, `password_hash`), via `@map`.
- **Enum names**: PascalCase (e.g. `TenantStatus`). Enum values are lowercase (e.g. `active`, `suspended`).
- **Constraint names**: `<table>_<column>_key` for unique, `<table>_<column>_idx` for index, `<table>_<column>_fkey` for foreign key. Composite constraints use `<table>_<col1>_<col2>_...`.

## 11.21 Migration risks

| Risk | Severity | Evidence |
|---|---|---|
| Composite FK added by raw SQL supplement | Low | The raw SQL is reviewed and committed. The migration is idempotent only against a freshly-Prisma-migrated database (per the migration comments). Re-running the migration against an arbitrary database state would fail. This is acceptable because Prisma's migration runner applies migrations in order. |
| No down migrations | Medium | Prisma migrations are forward-only. There is no `prisma migrate resolve --rolled-back` workflow. A bad migration requires a manual SQL rollback. The schema's `ON DELETE RESTRICT` everywhere means deleting seed data requires deleting in the correct order. |
| `@updatedAt` is auto-managed by Prisma | Low | If a future batch writes raw SQL that updates a row without going through Prisma, `updatedAt` will not be updated. The `SessionRepository` uses Prisma for all writes today. |
| No RLS policies | Medium | Tenant isolation is application-layer only. A SQL query run directly against the database can read any tenant's data. |
| `local_credentials.password_hash` is TEXT, not BYTEA | Low | The hash is a PHC-string-formatted Argon2id hash (ASCII). TEXT is appropriate. No risk. |
| `auth_sessions.token_hash` is CHAR(64) | Low | CHAR(64) is correct for a 64-character lowercase hex SHA-256 hash. CHAR pads to the declared length, but 64 chars is the exact length, so no padding occurs. No risk. |

## 11.22 Data-integrity risks

| Risk | Severity | Evidence |
|---|---|---|
| No soft-delete | Low (intentional) | The schema explicitly defers soft-delete to a future batch. Hard deletes are blocked by `ON DELETE RESTRICT` everywhere. |
| No audit trail for credential changes | Medium | `local_credentials.passwordChangedAt` is updated when the password is changed, but no audit record is written. The dev bootstrap script (`auth-bootstrap-dev.ts`) creates or updates the credential without any audit log. |

## 12.3 Configuration loading

- **API**: `ConfigModule.forRoot({ isGlobal: true, envFilePath: ['.env'], cache: true })`. This is the **deny-by-default** posture: only `.env` is read, only variables accessed via `configService.get<T>('KEY')` are exposed. `process.env` access from application code is used in three specific places (see below).
- **Direct `process.env` access** (intentional, documented in code comments):
  - `PrismaService` constructor reads `process.env['DATABASE_URL']` — to avoid a circular dependency with `ConfigModule`.
  - `PasswordService` constructor reads `process.env['ARGON2_MEMORY_COST']`, `ARGON2_TIME_COST`, `ARGON2_PARALLELISM` — same circular-dependency concern.
  - `auth.controller.ts` and `session-context.controller.ts` read `process.env['NODE_ENV']` for the cookie `secure` flag — to avoid coupling cookie behaviour to NestJS's config cache.
  - `openapi.setup.ts` reads `process.env.NODE_ENV` to suppress docs in production.
  - `auth-bootstrap-dev.ts` reads `process.env` for all its inputs (it's a standalone CLI, not a NestJS module).
- **Web**: `process.env.NEXT_PUBLIC_API_BASE_URL` is read at runtime in `getApiBaseUrl()`. No Next.js `.env` loading is configured explicitly; Next.js 16 automatically loads `.env`, `.env.local`, `.env.development`, `.env.production` per its documented precedence.

## 12.4 Configuration validation

- **`API_PORT`**: Validated in `main.ts`'s `readApiPort()` — throws if not a positive integer in 1–65535.
- **`WEB_ORIGIN`**: Validated in `main.ts`'s `readWebOrigin()` — throws if any entry does not match `/^https?:\/\//i`.
- **`NODE_ENV`**: Not validated. Any value other than `'production'` is treated as development.
- **`DATABASE_URL`**: Not validated at the application layer. Prisma validates the connection string when it tries to connect; an invalid URL produces a runtime error on the first query.
- **`ARGON2_*`**: Validated in `PasswordService`'s `readPositiveIntFromEnv()` — falls back to the default if the value is missing or non-numeric. **Soft-fail posture**, not a hard error.
- **`DEV_AUTH_*`**: Validated in `auth-bootstrap-dev.ts`'s `readBootstrapEnv()` — throws if any required variable is missing or if the password is not 12–128 characters.
- **`NEXT_PUBLIC_API_BASE_URL`**: Not validated. The `normaliseBaseUrl` helper trims and removes trailing slashes; an empty value falls back to the development default.

There is **no schema-validated configuration object** (no Zod schema for env vars, no `envalid` library, no `@nestjs/config` validation schema). Configuration validation is ad-hoc per variable.

- Cookie `secure: false` (so the cookie is sent over HTTP on localhost)
- OpenAPI docs served at `/api/docs` and `/api/docs-json`

## 12.6 Production configuration

Production requires (verified in code):
- `NODE_ENV` = `production` — enables cookie `secure: true`, suppresses OpenAPI docs, blocks the dev bootstrap.
- `DATABASE_URL` — a real PostgreSQL connection string (no default).
- `WEB_ORIGIN` — the production web origin (no default; falls back to localhost which would be wrong in production).
- `NEXT_PUBLIC_API_BASE_URL` — the production API base URL (no default; falls back to localhost which would be wrong in production).
- `API_PORT` — typically set by the platform (PORT env var on Heroku-like platforms; explicit port on Kubernetes).
- `ARGON2_*` — optionally overridden; defaults are conservative and production-safe.

## 12.7 CORS settings

Configured in `apps/api/src/main.ts`:
```ts
app.enableCors({
  origin: webOrigin,  // string or string[] from WEB_ORIGIN
  methods: ['GET', 'POST', 'PUT', 'PATCH', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Accept', 'X-CSRF-Token'],
  credentials: true,
});
```

- `origin` is **not** `true` (not reflective). It must be an exact match.
- `credentials: true` is required so the browser sends the HttpOnly cookie cross-origin (web on :3000, API on :3001 in dev).
- `X-CSRF-Token` is explicitly allowed so the preflight check does not reject logout or context-mutation requests.

## 12.8 Allowed origins

- The `WEB_ORIGIN` env var may be a single origin (`http://localhost:3000`) or a comma-separated list (`https://app.ibnhayan.example,https://admin.ibnhayan.example`).
- The list is parsed in `main.ts`'s `readWebOrigin()`: split on comma, trim, filter empty, validate each entry matches `/^https?:\/\//i`.
- A single-entry list returns a string; a multi-entry list returns a string array. NestJS's CORS handler accepts both.
- The auth and session-context controllers' `isOriginAllowed()` does an exact-match against the same value(s).

## 12.9 Cookie security behaviour

- `httpOnly: true` — always (JavaScript cannot read the cookie).
- `secure: true` in production, `false` in development (so the cookie is sent over HTTP on localhost).
- `sameSite: 'lax'` — always (cookie is sent on same-site navigations and same-origin requests, but NOT on cross-site POST requests — a CSRF defence in addition to the explicit CSRF token).
- `path: '/'` — always (cookie is sent to all API routes).
- `domain` — not set (cookie is bound to the exact origin that set it).
- `maxAge` — set to the session's remaining lifetime at login; set to `0` at logout to clear.

## 12.10 Database configuration

- **Provider**: PostgreSQL (declared in `schema.prisma`'s `datasource db { provider = "postgresql" }`).
- **Connection string**: read from `process.env.DATABASE_URL`. Not logged. Not persisted. Not surfaced in error messages.
- **Driver adapter**: `@prisma/adapter-pg` (`PrismaPg`). Required by Prisma 7.
- **Connection lifecycle**: lazy. The TCP connection is opened by the `pg` Pool on the first query. `PrismaService` does not call `$connect()` in `onModuleInit()` — this is the structural fix that lets the API boot and serve `/api/v1/health` without a database.
- **Disconnection**: `PrismaService.onModuleDestroy()` calls `$disconnect()`, which closes the `pg` Pool.
- **PostgreSQL version**: 17 (verified by `scripts/validate-database-foundation.sh`'s version check; the schema uses no PostgreSQL-17-specific features but the test bootstrap requires major version 17).
- **Extensions**: None. The schema comments say: "No PostgreSQL extensions are enabled. The schema uses only built-in PostgreSQL features."
- **RLS policies**: None. Explicitly deferred per the schema comments.
- **Database roles**: Single role (the one in `DATABASE_URL`). No per-tenant role separation.

## 12.11 Public frontend variables

Only one: `NEXT_PUBLIC_API_BASE_URL`. The `NEXT_PUBLIC_` prefix is required by Next.js to expose the value to the browser. No other `NEXT_PUBLIC_*` variable is referenced.

## 12.12 Missing environment validation

The following gaps exist:
1. **No Zod schema for env vars.** Configuration is validated ad-hoc per variable. A typo in `WEB_ORIGIN` (e.g. missing `http://`) would throw at API startup, but a typo in `ARGON2_MEMORY_COST` (e.g. `abc`) would silently fall back to the default.
2. **`.env.example` is incomplete.** It documents 5 of the 15 variables. The Argon2 parameters, the `ALLOW_DEV_AUTH_BOOTSTRAP` flag, and the `DEV_AUTH_*` variables are documented only in source-code comments.
3. **No production-readiness checklist.** There is no documented list of "must-set" variables for a production deployment. A future batch should add a production deployment runbook (the `13_DEPLOYMENT/PRODUCTION.md` file is a placeholder).

# 16. Exact Current User Flow

This section describes what a user can currently do, from opening the application until reaching the final implemented screen. Every step is verified against the actual code.

## 16.1 Landing-page flow

1. The user opens `http://localhost:3000/` in a browser.
2. Next.js renders `apps/web/src/app/page.tsx`, which renders `<LandingExperience />`.
3. `LandingExperience` composes `LandingHeader` + `Hero` + `OperationalClaritySection` + `SecuritySection` + `GrowthSection` + `LandingFooter`.
4. The `<html>` element has `lang="ar" dir="rtl"` (Arabic-first, RTL). The page renders in Arabic by default.
5. The `Hero` renders the hero title (`<h1>`), hero body, 4 value-indicator cards, and the `LoginPanel` (integrated into the hero).
6. The `LandingHeader` and `LandingFooter` each render a `LanguageSwitch` button. Clicking it toggles between Arabic and English. The `LanguageProvider` updates `document.documentElement.lang` and `dir` and re-renders all consuming components.
7. The `LoginPanel` runs a **silent session check** on mount: it calls `getSession()`. If a valid session exists (the user is already authenticated), the panel calls `router.replace('/dashboard')`. If no session exists (the common case), the form remains visible and the email input is focused for keyboard users.

## 16.2 Language selection

- The user clicks the `LanguageSwitch` button in the header or footer.
- The `LanguageProvider`'s `toggle()` function flips `lang` between `'ar'` and `'en'`.
- An `useEffect` updates `document.documentElement.lang` and `document.documentElement.dir`.
- All consuming components (which read `useLanguage()`) re-render with the new language's copy from `LANDING_COPY` (or `DASHBOARD_COPY` on the dashboard).
- The language choice is held in React memory only. It is **lost on hard reload**.

## 16.3 Login

8. The user enters their email and password in the `LoginPanel` form.
9. The user clicks "Sign in" (or presses Enter).
10. `LoginPanel.handleSubmit` calls `login({email, password})` from `apps/web/src/lib/api/auth/auth.client.ts`.
11. The client validates the input through `LoginRequestSchema.safeParse(input)`. If invalid, returns `{ok: false, error: contractInvalidError(...)}` without making a network request.
12. The client POSTs to `http://localhost:3001/api/v1/auth/login` with `credentials: 'include'`, `Content-Type: application/json`, `Accept: application/json`, `Origin: http://localhost:3000`, and the body `{email, password}`.
13. The API's `AuthController.login` parses the body through `LoginRequestSchema.safeParse`. If invalid, throws `invalidCredentials()` (401).
14. `AuthService.login` verifies the Origin header against `WEB_ORIGIN`. If disallowed, throws `originDisallowed()` (403).
15. `AuthService.login` looks up the user by `normalisedEmail`. If not found, throws `invalidCredentials()` (401).
16. If `user.status === 'disabled'`, throws `invalidCredentials()` (401).
17. `PasswordService.verifyForUser(user.id, password)` — Argon2id verification. If false, throws `invalidCredentials()` (401).
18. `AuthService.login` loads the user's memberships. If no `active` memberships, throws `invalidCredentials()` (401).
19. `AuthService.login` generates a 32-byte opaque token, hashes it with SHA-256, and persists the session row via `SessionRepository.create({userId, tokenHash, expiresAt, lastSeenAt})`.
20. `AuthController.login` sets the `ibn_hayan_session` HttpOnly cookie with the raw token.
21. `AuthController.login` returns the `SessionResponse` JSON body: `{user, memberships, activeTenantContext: null, expiresAt}`.
22. The web client's `login()` parses the response through `SessionResponseSchema.safeParse`. If valid, returns `{ok: true, data}`.
23. `LoginPanel.handleSubmit` calls `router.replace('/dashboard')`.

## 16.4 Session establishment

The session is now established. The browser holds the `ibn_hayan_session` HttpOnly cookie. The server holds the session row in `auth_sessions` with the SHA-256 hash of the token, the user ID, the expiry timestamp (now + 12h), the `lastSeenAt` (now), and `activeTenantMembershipId = null` (no context selected yet).

## 16.5 Tenant selection

24. The user lands on `/dashboard`. `DashboardPage` mounts and calls `getSession()` to verify the session. On success, the session response is stored in React state.
25. `DashboardPage` calls `getContext()` → `GET /api/v1/context`. The response is `{options: [...memberships], active: null}` (no context selected yet).
26. The dashboard renders a workspace card with:
   - The current context (empty — "No workspace selected").
   - A radio group of available workspaces (the user's active memberships under active tenants).
   - A "Select / Change" button and (when an active context exists) a "Clear selection" button.
27. The user selects a workspace from the radio group. `selectedMembershipId` is updated in React state.
28. The user clicks "Select / Change". `DashboardPage.handleSelectContext` calls `getCsrfToken()` → `GET /api/v1/auth/csrf`. The API returns `{token: "..."}`. The token is held in component memory only.
29. `DashboardPage.handleSelectContext` calls `selectTenantContext(selectedMembershipId, token)` → `PUT /api/v1/context/tenant` with the membership ID in the body and the CSRF token in the `X-CSRF-Token` header.
30. `SessionContextController.selectTenantContext` verifies Origin, parses the body, authenticates the session, verifies the CSRF token, and delegates to `SessionContextService.selectContext`.
31. `SessionContextService.selectContext` verifies the membership exists, belongs to the user, is active, and belongs to an active tenant. If any check fails, throws `contextSelectionForbidden()` (403). Otherwise, persists `activeTenantMembershipId` on the session row.
32. The API returns the updated `ContextResponse`: `{options: [...], active: {membershipId, tenantId, tenantSlug, tenantDisplayName}}`.
33. The dashboard updates its React state. The workspace card now shows the selected workspace as the current context.

## 16.6 Dashboard access

The user is now on the dashboard with:
- A sticky header showing the brand mark, the active-workspace chip (showing the selected tenant's display name), the language switch, and the logout button.
- A page title (`<h1>`) and subtitle.
- A workspace card with the current context and the radio group (the selected workspace is highlighted).
- An account card with the user's display name, email, session-status chip, and the list of available workspaces (memberships).

The user can:
- Switch workspace by selecting a different radio and clicking "Select / Change".
- Clear the workspace by clicking "Clear selection" (requires a fresh CSRF token, which the dashboard fetches automatically).
- Toggle language between Arabic and English.
- Sign out.

## 16.7 Logout

34. The user clicks "Sign out" in the dashboard header.
35. `DashboardPage.handleLogout` calls `getCsrfToken()` → `GET /api/v1/auth/csrf`. The API returns a fresh token.
36. `DashboardPage.handleLogout` calls `logout(token)` → `POST /api/v1/auth/logout` with the CSRF token in the `X-CSRF-Token` header.
37. `AuthController.logout` verifies Origin, validates the cookie, looks up the session, verifies the CSRF token, revokes the session (sets `revokedAt = now`), and invalidates the CSRF token in `CsrfService`.
38. `AuthController.logout` clears the `ibn_hayan_session` cookie via `res.clearCookie`.
39. The API returns `{ok: true}`.
40. `DashboardPage.handleLogout` calls `router.replace('/login')`. The user is back on the landing page.

## 16.8 Failure scenarios

| Scenario | What happens |
|---|---|
| Wrong password at login | API returns 401 with `{error: {code: 'AUTH_INVALID_CREDENTIALS', message: 'Invalid email or password.'}}`. The `LoginPanel` displays "Invalid email or password." (or the Arabic equivalent). The form remains visible. |
| Unknown email at login | Same as wrong password — the API returns the same generic 401. The user cannot tell whether the email is unknown or the password is wrong. |
| Disabled user at login | Same as wrong password. |
| User with no active membership at login | Same as wrong password. |
| Login throttled (10+ attempts/60s/IP) | API returns 429. The `LoginPanel` displays "An unexpected error occurred. Please try again." (or the Arabic equivalent). |
| Missing or disallowed Origin at login | API returns 403 with `{error: {code: 'AUTH_ORIGIN_DISALLOWED'}}`. The `LoginPanel` displays a generic error. |
| Network error (API unreachable) | The client returns `{ok: false, error: {category: 'NETWORK_ERROR'}}`. The `LoginPanel` displays "Unable to reach the server. Please try again later." |
| Session expired (12h after login) | The next `getSession()` call returns 401. On the dashboard, `DashboardPage` redirects to `/login`. On the landing page, the silent session check fails silently and the form remains visible. |
| Session revoked (e.g. user signed out from another device — not currently implemented; would require a `revokeAllSessionsForUser` endpoint) | Same as expired session. |
| CSRF token invalid on logout | API returns 403 with `{error: {code: 'AUTH_CSRF_INVALID'}}`. The `DashboardPage` displays "Unable to sign out. Please try again." |
| Cross-user context selection (membership belongs to another user) | API returns 403 with `{error: {code: 'CONTEXT_SELECTION_FORBIDDEN', message: 'The selected tenant context is not available.'}}`. The dashboard displays "Unable to update the workspace. Please try again." The response does **not** reveal whether the membership exists for another user. |
| Database unreachable | The API boots (Prisma connects lazily). The first auth request fails with a Prisma connection error. NestJS's default exception filter returns 500. The client displays a generic error. |

- No inventory screen.

- No reports.

- No tenant-management screen.

The user can sign out, switch workspaces, clear the workspace, and toggle the language. That is the entire implemented surface.

# 17. Recommended Next Batch

## Batch Name

**Batch 7 — Role/Permission Catalogue and Authorization Guard**

## Objective

Introduce a minimal role/permission catalogue on `TenantMembership` and a NestJS authorization guard that consumes it. This is the prerequisite for every subsequent tenant-scoped business module (patients, appointments, etc.), because those modules need to ask "is this user allowed to perform this action in this tenant?" before doing anything.

## Scope

1. Add a `role` enum to `TenantMembership` (start with 3 values: `owner`, `member`, `viewer` — see "Out of Scope" for why this is minimal).
2. Add a Prisma migration that introduces the `role` column with a default of `member` for existing rows.
3. Update the `TenantMembership` domain type in `packages/domain/src/identity/membership.ts` to carry the role.
4. Update the `TenantMembershipSummary` contract in `packages/contracts/src/auth/auth.schema.ts` to expose the role.
5. Update `AuthService.buildSessionResponse` to include the role in the membership summary.
6. Define a minimal permission catalogue: which roles can perform which actions. Start with 3 actions: `context:select`, `context:clear`, `context:view`. (These are the only actions that exist today.)
7. Introduce an `AuthorizationService` in `apps/api/src/modules/auth/` that, given a session and an action, returns `true`/`false`.
8. Introduce an `@RequirePermission(action)` decorator and an `AuthorizationGuard` that consults the `AuthorizationService`. The guard depends on `AuthService` (to load the session) and on the active Tenant context (to load the user's role for that tenant).
9. Update the `SessionContextController` to use the guard on `PUT /context/tenant` and `DELETE /context/tenant` (with `context:select` and `context:clear` respectively).
10. Update the dashboard to display the user's role for the active workspace.

## Out of Scope

- **A full RBAC engine** with fine-grained permissions, permission groups, role inheritance, and dynamic role assignment. The catalogue starts with 3 roles and 3 actions. Future batches can extend it.
- **A role-management UI**. Roles are assigned via the dev bootstrap script or via SQL in this batch. A role-management UI is a future batch.
- **Audit logging of authorization decisions.** The audit primitive is a separate (Critical) batch.
- **A permission catalogue for business modules** (patients, appointments, etc.). Those permissions will be defined alongside their respective modules.
- **Multi-role memberships.** A user has exactly one role per membership in this batch. Multi-role (e.g. "member + billing-admin") is a future batch.
